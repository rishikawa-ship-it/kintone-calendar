## §1 背景

実機で以下のエラーが発生している。

```
[KC.RenderMonth] loadEvents error: {
  code: 'GAIA_TM12',
  message: '作成できるカーソルの上限に達しているため、カーソルを作成できません。
            不要なカーソルを削除するか、しばらく経ってから再実行してください。'
}
```

kintone は 1 ドメインあたり同時に作成できるカーソルの上限が **10 個** に制限されている。
カーソルが正常に削除されないまま残留すると、上限に達して後続の取得処理が全て失敗する。
現状エラーは `console.error` のみで、ユーザーにはカレンダーが空白表示されるだけでありエラーの原因が分からない。

---

## §2 現状分析

### §2.1 カーソル使用箇所

`plugin/src/js/desktop.js` の `KC.Api.loadEvents`（行 692〜791）がカーソル API を使用している。

- カーソル作成: 行 763 `kintone.api(cursorUrl, 'POST', createParams)`
- カーソル GET ループ: 行 769 `kintone.api(cursorUrl, 'GET', { id: cursorId })`
- エラー時カーソル DELETE: 行 783 `kintone.api(cursorUrl, 'DELETE', { id: cursorId })`

カーソル URL は `kintone.api.url('/k/v1/records/cursor.json', true)` で構築（行 752）。

### §2.2 カーソル削除の実装状況

`loadEvents` は以下の構造になっている（行 761〜790）:

```
cursorId = null
try
  POST /records/cursor.json → cursorId 取得
  while (hasNext)
    GET /records/cursor.json → レコード取得
    hasNext が false になった時点でカーソルは kintone により自動削除される
catch (err)
  if (cursorId) DELETE /records/cursor.json  ← エラー時のクリーンアップ
  console.error(...)
  throw err
```

**設計上の問題点（根本原因）**: カーソル作成直後（POST 成功後・GET 前）、または GET ループの途中で例外が発生した場合、catch 節で DELETE を試みる実装になっている。この点は適切に見える。

しかし **GAIA_TM12 は「カーソル作成時（POST）に上限超過」で発生するエラー**であり、DELETE の問題ではなく「既に 10 個のカーソルが残留している」状態を示す。

残留カーソルが蓄積する経路として、次のシナリオが疑われる:

1. **ビュー切替の高速連打**: ユーザーが月/週/日ビューを素早く切り替えると、前の `loadEvents` の非同期処理が完了する前に次の `loadEvents` が起動される。前の処理が途中で打ち切られた場合（JavaScript は単一スレッドだが `await` の間に別の処理が割り込む）、前のカーソルが残留する可能性がある。
2. **ブラウザのページリロード / タブ移動**: カーソル GET ループの途中でリロードされると、サーバー側にカーソルが残る（残留カーソルは約 10 分で自動失効）。
3. **プラグイン初回アップロード後の連続テスト**: 開発・検証時に短時間でリロードを繰り返すことで残留カーソルが蓄積する。

### §2.3 loadEvents の呼び出し箇所

`loadEvents` は以下の 3 箇所から呼ばれている:

| 呼び出し元 | 行 | ビュー |
|---|---|---|
| `KC.Render.refresh` | 行 3575 | 週ビュー（デフォルト） |
| `KC.RenderMonth` の `refresh` | 行 4378 | 月ビュー |
| `KC.RenderDay` の `refresh` | 行 4880 | 日ビュー |

3 か所とも catch 節は `console.error` のみで、ユーザーへの通知がない。

### §2.4 レコード件数の評価

検証アプリのレコード件数は 4 件程度。カーソル API（1 回あたり最大 500 件）は通常 GET `/records.json`（最大 500 件）で代替可能な規模である。カーソル API が絶対必要なのは 500 件超のケースのみ。

現状のカーソル使用は `kintone-rules.md §3`「大量レコード取得時はカーソル API を使用する」に準拠した設計だが、実際には大量データが存在しないため、通常 GET への切替も現実的な選択肢となる。

### §2.5 エラー通知の現状

| 種別 | 現状 |
|---|---|
| DnD 更新失敗 | `alert('予定の更新に失敗しました: ' + msg)` （行 1328）|
| loadEvents 失敗 | `console.error` のみ（行 3579, 4388, 4884）|
| カーソル DELETE 失敗 | `console.error` のみ（行 785）|
| CSS/JS 読み込み失敗等 | `console.error` のみ |

HANDOVER.md §8「API 失敗通知: 現状は `alert()` 据え置き。後フェーズで UI 化予定」と記録されている。loadEvents の失敗は alert もなく完全にサイレント。

## §3 機能要件

### FR-1: カーソル上限エラーの根本対策（**確定: 案 B — 通常 GET 切替**）

カーソル API を廃止し、表示期間で絞り込んだ `GET /records.json`（最大 500 件）に変更する。

#### FR-1（確定内容）

- 月ビュー・週ビュー・日ビューの全 `loadEvents` 呼び出しを対象とする
- カーソル API（`POST /records/cursor.json`、`GET /records/cursor.json`、`DELETE /records/cursor.json`）を全廃する
- 代替 API: `GET /records.json` に `offset=0&limit=500` を指定して 1 リクエストで取得する
- クエリ条件（表示期間による絞り込み・既存フィルタ）は現行のまま維持する
- 実装後、コード内に `/records/cursor` の文字列が残っていないことを確認する（AC-1 参照）

#### FR-1 の制約・前提

- 運用アプリの表示期間内レコード件数が 500 件以内であることを前提とする
- 500 件を超えた場合の挙動は §7 未確定事項 Q1 参照

### FR-2: エラー時のユーザー通知 UI（**確定: バナー方式**）

`loadEvents` 失敗時にカレンダー上部固定のバナーでユーザーへ視覚的なエラー通知を行う。

#### FR-2（確定内容）

- **表示位置**: カレンダー上部に固定表示（`.kc-calendar-root` の最上部に挿入）
- **内容**: エラーメッセージ + 「再読み込み」ボタン
- **配色**: 警告色（背景 `#fff3cd`（黄系）または `#f8d7da`（赤系））、テキスト黒
- **エラーメッセージの分岐**:
  - `err.code === 'GAIA_TM12'` の場合: 「カーソルの上限に達しました。しばらく待ってから再読み込みしてください。」
  - その他の API エラー: 「データの読み込みに失敗しました。再読み込みしてください。」
- **再読み込みボタン**: クリックで `KC.Render.refresh()`（または同等の `refresh()` 関数）を再実行し、成功時にバナーを自動消去する
- **閉じる手段**: 再読み込み成功時に自動消去。×ボタンによる手動閉じの要否は §7 未確定事項 Q2 参照

## §4 受入基準

### AC-1: カーソル API の完全廃止

**Given** 実装完了後のソースコード（`plugin/src/js/desktop.js`）において
**When** `/records/cursor` で grep を実行したとき
**Then** マッチ件数がゼロである

### AC-2: 通常 GET による全件取得

**Given** 運用アプリの表示期間内レコード件数が 500 件以内の場合
**When** `loadEvents` を実行したとき
**Then** `GET /records.json` が 1 リクエスト発行され、表示期間内の全レコードが取得できる

### AC-3: 500 件超過時のバナー通知

**Given** 表示期間内のレコード件数が 500 件を超える場合
**When** `GET /records.json` が 500 件を返したとき（= レコードが切り捨てられた可能性がある）
**Then** バナーに「表示件数が上限（500 件）に達しています。一部の予定が表示されていない可能性があります。」に相当するメッセージを表示する

> 注: 500 件超過の検知方法は「レスポンスの `records.length === 500` かつクエリ条件に件数上限がない場合に警告」とする実装を推奨する。自動ページングは本 REQ スコープ外。

### AC-4: loadEvents エラー時のバナー表示

**Given** `loadEvents` が API エラーで失敗した場合
**When** エラーが catch された後
**Then** カレンダー上部にエラーバナーが表示され、「再読み込み」ボタンが含まれる

### AC-5: 再読み込みボタンによる再取得

**Given** エラーバナーが表示されている状態で
**When** ユーザーが「再読み込み」ボタンをクリックし、API が正常応答した場合
**Then** バナーが消去され、カレンダーが正常表示される

### AC-6: GAIA_TM12 専用メッセージ

**Given** kintone から `code: 'GAIA_TM12'` のエラーが返った場合
**When** エラーバナーが表示されるとき
**Then** 「カーソルの上限に達しました。しばらく待ってから再読み込みしてください。」に相当するメッセージが表示される

### AC-7: 通常動作への影響なし

**Given** API が正常に応答する状態で
**When** カレンダーを表示するとき
**Then** イベントが従来通り表示され、エラーバナーは表示されない

## §5 設計案比較（参考記録）

### §5.1 カーソル対策の案比較

| 案 | 概要 | メリット | デメリット | 採否 |
|---|---|---|---|---|
|**A: 重複リクエスト制御**| 進行中の loadEvents を検知し、古いカーソルを DELETE してから新リクエスト開始 | 根本的な残留防止。カーソル API は維持 | 実装が増える | **不採用** |
|**B: 通常 GET 切替**| カーソル API を廃止し GET /records.json に変更 | シンプル・カーソルリーク不要 | 500 件超で漏れが生じる | **採用（確定）** |
|**C: try-finally 強化**| catch → try-finally に変更し、失敗時の DELETE をより確実にする | 既存構造に近く変更最小 | 高速連打による同時並行リクエストは防げない | **不採用** |

案 B 採用理由: 現在の検証アプリはレコード 4 件程度でカーソル API が必要な規模ではなく、カーソル API を廃止することで残留問題を構造的に解消できる。`GET /records.json` は 1 リクエストで完結し保守性が高い。

### §5.2 エラー通知 UI の案比較

| 案 | 実装方法 | 自動消去 | 再試行 UI | 実装コスト | 採否 |
|---|---|---|---|---|---|
|**トースト**| カレンダー右上に数秒表示、自動消去 | あり（3〜5 秒） | 任意追加可 | 低〜中 | **不採用** |
|**バナー**| カレンダー上部に固定表示（再取得成功で消える） | 再取得成功時に自動消去 | ボタン配置しやすい | 中 | **採用（確定）** |
|**モーダル**| ページ全体をブロックするダイアログ | なし | あり | 中 | **不採用** |

バナー採用理由: カレンダー全体が空白になる重大エラーであり、自動消去なし・再読み込みボタン付きのバナーがこの重度に適合する。トーストは軽微なエラー向き。モーダルはインタラクションを過剰にブロックする。

## §6 採用案（確定）

### カーソル対策: 案 B（通常 GET 切替）— 確定

- カーソル API 全廃。`GET /records.json?app=...&query=...&fields=...&offset=0&limit=500` に変更
- 月/週/日ビューの `loadEvents` を全て対象とする
- 既存のクエリ条件（表示期間絞り込み・フィールド指定）は維持する

### 通知 UI: バナー方式 — 確定

- 配置: `.kc-calendar-root` の最上部に `<div class="kc-error-banner">` を挿入
- 内容: エラーメッセージ + 「再読み込み」ボタン（`KC.Render.refresh()` または各ビューの `refresh()` を呼ぶ）
- スタイル: 背景 `#fff3cd`（警告黄）または `#f8d7da`（エラー赤）、テキスト黒
- 消去条件: 再読み込み成功時に自動消去

## §9 想定 UX / シーケンス

### 通常 GET 切替後の loadEvents フロー（案 B — 確定）

```
ユーザー: ビュー表示 / 月切替
  ↓
KC.Render.refresh() / KC.RenderMonth.refresh() / KC.RenderDay.refresh()
  ↓
KC.Api.loadEvents(isoStart, isoEnd)
  ↓
GET /records.json?app=...&query=...&fields=...&offset=0&limit=500
  ↓（成功）
records.length === 500 → 超過警告バナー表示（Q1 確定後に対応）
records.length < 500  → バナーなし
  ↓
レコード配列 → _recordToEvent() → S.events
  ↓
renderGrid() → placeEvents() → カレンダー描画
```

### エラー時のバナー表示フロー（確定）

```
KC.Api.loadEvents()
  ↓（失敗）
catch (err)
  ↓
err.code === 'GAIA_TM12' → 専用メッセージ生成
その他 → 汎用メッセージ生成
  ↓
showErrorBanner(message) → DOM にバナー挿入（カレンダー上部固定）
  ↓
ユーザー: 「再読み込み」ボタンをクリック
  ↓
hideErrorBanner() → KC.Render.refresh() 再実行
  ↓（成功）
バナー消去 → カレンダー正常表示
```

## §10 前提・スコープ外

- 本 REQ はカーソル API の通常 GET 切替と loadEvents エラー通知 UI の追加を対象とする
- DnD 更新失敗の alert（行 1328）の UI 改善は本 REQ スコープ外とする（別途起票推奨）
- 500 件超レコード対応の完全な自動ページング処理は本 REQ スコープ外とする（Q1 確定後に別途検討）
- kintone 側のカーソル手動削除（管理者操作）は本 REQ の対象外（即時の回避策として案内可）
