- データ実態（アプリ全体設定 or `views[viewId]`）に合わせて「アプリ全体設定」と「ビュー個別設定」を正しく分離する。

## 1. 背景・目的

`REVIEW_config-screen.md` の評価により、現状の設定画面には以下の構造課題が確認されている（詳細は同レポート §2, §3 参照）。

- 「ビュー個別設定」という見出しの中に、実データ上はアプリ全体設定である 4 ブロック（権限フィールド設定・権限ユーザー設定・デフォルト権限設定・フィルタ設定）が閉じ込められている（`REVIEW_config-screen.md` §2.2, §3.1）。
- 「共通設定（フィールドマッピング）」に、テーマの異なる「必須/任意フィールド」と「重複チェック設定」が同居している（同 §2.2 問題点2）。
- デフォルト権限設定が「権限ユーザー設定」カードの内部にネストされており、「両テーブル横断のフォールバック」という性質が伝わりにくい（同 §5.1）。
- 「検索対象フィールド設定」が独立セクション化されている基準が不明瞭（同 §2.2 問題点3）。

本書は、評価レポートで提示された 3 案（案A: セクション再編 / 案B: タブ・アコーディオン化 / 案C: 共通コンポーネント化）のうち **案A（見た目の再編・セクション再グルーピング）** を選択した上での詳細設計を行う。実装は行わない。

## 2. 現状分析

### 2.1 アプリ全体設定 vs ビュー個別設定の分類確定（コード裏取り済み）

`plugin/src/js/config.js` のトップレベル `currentConfig` 構造（`plugin/src/js/config.js:160-186`）と `saveConfig()` 内の `finalConfig` 組み立て（`plugin/src/js/config.js:3490-3501`）を正本として分類する。

| 項目（config.html 上の見出し） | 保存キー | 粒度 | 裏取り（ファイル:行） |
|---|---|---|---|
| 必須/任意フィールド（タイトル/開始/終了/終日/色/場所/メール/メモ） | `fieldMapping.{fieldTitle, fieldStart, ...}` | **アプリ全体** | `config.js:8`, `collectFieldMapping()` 定義付近 `config.js:561-607`、`applyFieldMapping()` `config.js:2373-2424` |
| 重複チェック設定（判定モード・キー/参照テーブル/リソース識別フィールド） | `fieldMapping.{overlapMode, overlapKeyFieldCode, ...}` | **アプリ全体** | `config.js:9-16` |
| 検索対象フィールド設定 | `searchTargets: [{fieldCode}]` | **アプリ全体** | `config.js:19`, `collectSearchTargets()` `config.js:2329`, `applySearchTargets()` `config.js:2344` |
| カレンダータイトル・初期ビュー | `views[viewId].{calendarTitle, defaultView}` | **ビュー個別** | `collectViewConfig()` `config.js:2357-2363`（`elCalendarTitle.value` / `defaultView` チェック済みラジオを取得）、`applyViewConfig(viewId)` `config.js:2494-2536`（`currentConfig.views[viewId]` を参照して DOM に反映） |
| 権限フィールド設定 | `fieldValueRules: [...]` | **アプリ全体** | `config.js:18`, `collectFieldValueRules()` `config.js:1590`、`applyFieldValueRules()` `config.js:1707`。`init()` 内で `applyFieldValueRules()` はビュー選択と無関係に一度だけ呼ばれる（`config.js:3675`） |
| 権限ユーザー設定 | `permissionRules: [...]` | **アプリ全体** | `config.js:17`, `collectPermissionRules()` `config.js:1304`、`applyPermissionRules()` `config.js:1335`。`init()` 内で一度だけ呼ばれる（`config.js:3671`） |
| デフォルト権限設定 | `defaultPermission: {enabled, permission, bgColor, textColor}` | **アプリ全体** | `config.js:158`（型定義コメント）、`collectDefaultPermission()` `config.js:1687`、`applyDefaultPermission()` `config.js:1646`。`init()` 内で一度だけ呼ばれる（`config.js:3693`） |
| フィルタ設定 | `filterConfig: {groups: [...]}` | **アプリ全体** | `config.js:20`, `collectFilterConfig()` `config.js:2168`、`applyFilterConfig()` `config.js:2223`。`init()` 内で一度だけ呼ばれる（`config.js:3679`） |

**結論**: `views[viewId]` として実際にビュー単位で保存されるのは **「カレンダータイトル」「初期ビュー」の 2 項目のみ**。それ以外はすべてアプリ全体設定である。`saveConfig()` 内の `currentConfig.views[currentViewId] = collectViewConfig()`（`config.js:3382`）が `views[viewId]` に書き込む唯一の経路であり、`collectViewConfig()` の返り値が `{calendarTitle, defaultView}` の 2 キーのみであることからも、ビュー個別設定の実データ範囲はこの 2 項目に限定されることが確定する。

また、`handleViewSelectChange()`（ビュー切替時のハンドラ）が呼ぶのは `applyViewConfig(viewId)` のみであり、`applyPermissionRules` / `applyFieldValueRules` / `applyDefaultPermission` / `applyFilterConfig` はビュー切替時に再呼び出しされない（`init()` 内で 1 回のみ）。これは「ビューを切り替えても権限・フィルタ設定は変化しない」という実装事実の直接的な裏付けである。

### 2.2 補足: 検索対象フィールド設定について

`searchTargets` はアプリ全体設定だが、`fieldMapping` とは別トップレベルキーで保存されている（`config.js:2724` 付近の `loadInitialConfig()` 初期値、`config.js:3495`）。キーが分かれている実装理由は歴史的経緯（v7 で追加）であり、UI 上は「アプリ全体設定」という同じ性質のグループに含めることに支障はない（保存キーは変更しないため、`collectSearchTargets()` / `applySearchTargets()` の呼び出し元・呼び出しタイミングはそのまま）。

### 2.3 現状の DOM 構造と ID 依存

- すべての DOM 要素取得は `document.getElementById(...)` または `document.querySelector('#id ...')` によるフラットな ID 参照であり、親要素の階層・兄弟関係には依存していない（`config.js:238-307` の `elXxx` 変数群を確認。約 30 個の `getElementById` 呼び出しがすべて ID 直接指定）。
- 行 DnD（`attachRowDragEvents()`, `config.js:1067-1148`）は `row.parentNode`（= 各テーブルの `rows` コンテナ、例: `#kc-permission-rows`）を基準に `insertBefore` するため、**コンテナ自身がどのセクション・カードの中に置かれているか** は DnD ロジックに影響しない。
- カラーピッカー挿入（`init()` 内 `config.js:3683-3690`）は `document.querySelector('#kc-default-permission-fields .kc-default-permission-col-bgcolor')` のように ID 起点の相対検索であり、`#kc-default-permission-fields` という ID さえ保持されれば、その親カードをどこに移動しても動作に影響しない。

**総合評価**: 現状の実装は ID 参照ベースであり、**DOM のセクション再配置（親要素の移動）そのものは低リスク**。壊れやすいのは「ID を変更してしまう」「id 付き要素を消してしまう」「同じ id を重複させる」場合のみである。

## 3. 要件

### 3.1 新セクション構成案

以下の 5 セクション構成に再編する。見出しレベルは `<h2 class="kc-config-section-title">` を各セクションの最上位に置く（現状の `<section class="kc-config-section">` の粒度を維持しつつ、内容を移動する）。

```
[セクション1] 基本設定（フィールドマッピング）
    └─ 必須フィールド（タイトル/開始/終了）
    └─ 任意フィールド（終日/色/場所/メール/メモ）

[セクション2] 表示・絞り込み
    └─ 検索対象フィールド設定
    └─ フィルタ設定（チェックボックスパネル）

[セクション3] 権限・色分け
    └─ 権限フィールド設定（値ベース、優先度高）
    └─ 権限ユーザー設定（USER_SELECT ベース）
    └─ デフォルト権限設定（★独立カード化。両テーブル横断のフォールバック）

[セクション4] 重複チェック設定
    └─ 判定モード（無効/モードA/モードB）
    └─ モードA設定エリア（条件表示）
    └─ モードB設定エリア（条件表示）

[セクション5] ビュー個別設定
    └─ 対象ビューの選択（制御パネル）
    └─ 選択中ビューの設定（カレンダータイトル・初期ビューのみ）
```

**再編の考え方**:
- セクション1（基本設定）は現状の「共通設定」から重複チェック設定を分離し、フィールドマッピングという単一テーマに揃える。
- セクション2（表示・絞り込み）は「ユーザーがカレンダー閲覧時に使う絞り込み系」という利用者視点のテーマでまとめる。検索対象フィールド設定とフィルタ設定は保存キーは異なるが、いずれも「カレンダーヘッダーの検索/フィルタ UI に表示する項目を管理者が定義する」という共通の目的を持つため同一セクションが妥当。
- セクション3（権限・色分け）は判定順序（フィールド→ユーザー→デフォルト）を上から下へ配置し、デフォルト権限設定を独立カード化する（§3.3 参照）。
- セクション4（重複チェック設定）はテーマが独立しているため単独セクションに昇格。REVIEW §2.2 問題点2 の解消。
- セクション5（ビュー個別設定）は真にビュー単位のデータ（カレンダータイトル・初期ビュー）のみを残し、「ビュー個別設定」という見出しが指す内容とデータの実態を一致させる（REVIEW §2.2 問題点1の解消）。対象ビューの選択（制御パネル）はこのセクションの前提操作として同セクション内に残す。

### 3.3 デフォルト権限設定の独立表示

**現状の課題**（REVIEW §5.1, §4.5）: `.kc-default-permission-section` は `.kc-permission-section`（権限ユーザー設定カード）の内部にネストされ、「権限ユーザー設定の一部機能」に見える。実際は権限フィールド設定・権限ユーザー設定の**両方**を通じたフォールバックである。

**再編方針**:
- HTML 上、`.kc-default-permission-section`（`config.html:402-433`）を `.kc-permission-section`（権限ユーザー設定カード, `config.html:373`）の外に出し、権限フィールド設定カード・権限ユーザー設定カードと**並列の第3カード**として配置する。
- 配置順序: 「権限フィールド設定」→「権限ユーザー設定」→「デフォルト権限設定」（判定優先順位と一致させる）。
- 見出し文言案: `デフォルト権限設定（両方の設定に一致しない場合）`
- 補足文（ヘルプ）案:
  > 「権限フィールド設定」「権限ユーザー設定」のどちらにも一致しないレコードに適用される、共通のフォールバック設定です。チェックを外すと、従来の動作（設定あり非マッチ時は閲覧のみ／設定なし時は全員編集可）を維持します。
- 視覚的な扱い: 3 カードを同じ見た目の強さ（背景色・枠線）で並べ、「対等な第3の判定ルール」であることを示す。現状のような「カード内カード（ネスト）」構造は解消する。
- 依存関係の見せ方: 「デフォルト設定を使用する」チェックボックス ON 時のみ権限セレクト・カラーピッカーを有効化する現行の disabled 制御（`_syncDefaultPermissionEnabled()`, `config.js:1632`）は維持する。

**具体的な HTML 変更の性質**: すべて「既存の `<div>` ブロックをそのまま別の親 `<section>` に移す」というカット&ペーストであり、ブロック内部のマークアップ（`id`/`name`/`class`）は変更しない。新設が必要なのは以下のみ。

### 3.6 実装への影響（config.js の init 挿入順・イベント登録箇所）

- **DOM 要素取得（`elXxx = document.getElementById(...)`）**: 変更不要。すべて ID 参照であり、ID を変更しない限り取得先要素の親構造が変わっても動作に影響しない（§2.3 参照）。
- **`init()` の呼び出し順序（`config.js:3648-3901`）**: 変更不要。`applyFieldMapping` → `applyPermissionRules` → `applyFieldValueRules` → `applyFilterConfig` → デフォルト権限カラーピッカー挿入 → `applyDefaultPermission` → `applySearchTargets` → `refreshPerViewSelect` という順序は、DOM 上の新しい配置順（セクション1→2→3→4→5）と完全一致しないが、**JS の実行順序と HTML の視覚的な配置順は独立**であり、順序を合わせる必要はない（ID 参照ベースのため）。ただし可読性のため、DOM 変更と合わせて `init()` 内のコメント（「3.5」「3.6」等のステップ番号）を新セクション順に振り直すことを推奨する（任意、必須ではない）。
- **カラーピッカー挿入箇所（`config.js:3683-3690`）**: `document.querySelector('#kc-default-permission-fields .kc-default-permission-col-bgcolor')` という ID 起点のクエリのため、`#kc-default-permission-fields` を含む親カードをどこに移動しても影響なし。**ただし `#kc-default-permission-section` を `.kc-permission-section` の外に出す際、この ID 自体を保持すること**が唯一の制約。
- **イベント登録箇セ所（`config.js:3786-3816`）**: `elPermissionAdd` / `elFieldValueAdd` / `elSearchTargetAdd` / `elFilterGroupAdd` の `click` リスナー登録はすべて ID 参照のボタン要素に対して行われるため、ボタンの親カードの移動による影響はない。
- **`KC.CheckFilter.init` / `KC.ShowAllToggle`**: これらは `plugin/src/js/desktop.js`（カレンダー本体、カレンダーヘッダーへのフィルタボタン・全表示トグルボタンの挿入処理。`desktop.js:10461,10465` 等）に属する処理であり、**設定画面（config.js）とは無関係**。本件（config 画面再編）の影響範囲外であることを確認済み。

### 3.7 マイグレーション不要の確認

- `finalConfig`（`config.js:3490-3501`）のキー構成・`version: 11`（`config.js:3491`）は本件で変更しない。
- `loadInitialConfig()` 内の v2〜v11 マイグレーション連鎖（`config.js:2645-2843` 付近）はキー名・階層構造に対する処理であり、DOM 上の見た目の並びとは無関係。本件による HTML/CSS の再配置は `loadInitialConfig()` に一切影響しない。
- 既存ユーザーが保存済みの config データ（version 11 形式）は、再編後の画面でも `applyXxx()` 系関数がそのまま読み込む。**マイグレーション処理の追加・変更は不要**。

## 4. 受入基準

Given/When/Then 形式で記述する。

### AC-1: セクション構成の再編

- **Given** 再編後の `config.html` を開く
- **When** ページを上から下にスクロールする
- **Then** セクション見出しが「基本設定（フィールドマッピング）」→「表示・絞り込み」→「権限・色分け」→「重複チェック設定」→「ビュー個別設定」の順に表示される

### AC-2: 「ビュー個別設定」の内容がビュー個別データのみになる

- **Given** 再編後の画面で「ビュー個別設定」セクションを見る
- **When** セクション内の項目を確認する
- **Then** 表示される項目は「対象ビューの選択」「カレンダータイトル」「初期ビュー」のみであり、権限・フィルタ関連の項目は存在しない

### AC-3: ビュー切替時に権限・フィルタ設定が変化しないことが画面構造からも自然に理解できる

- **Given** 再編後の画面で、権限・色分けセクションに何らかのルールを設定した状態
- **When** 「ビュー個別設定」セクションでビュー選択プルダウインを別のビューに切り替える
- **Then** 「権限・色分け」セクションの内容は変化しない（この挙動自体は既存仕様のまま。画面構造上「権限・色分け」が「ビュー個別設定」の外にあることで、ビューに紐づく設定ではないと視覚的に判別できる）

### AC-4: デフォルト権限設定の独立カード化

- **Given** 再編後の「権限・色分け」セクションを見る
- **When** 権限フィールド設定・権限ユーザー設定・デフォルト権限設定の3カードを確認する
- **Then** デフォルト権限設定が権限ユーザー設定カードの内部ではなく、独立した並列カードとして表示される
- **And** デフォルト権限設定カードの見出し直下に「両方の設定に一致しない場合」であることを示す補足文が表示される

### AC-5: 重複チェック設定の独立セクション化

- **Given** 再編後の画面を確認する
- **When** 「基本設定（フィールドマッピング）」セクションの内容を見る
- **Then** 重複チェック設定（判定モードラジオ・モードA/B設定エリア）は表示されず、別の独立セクション「重複チェック設定」に表示される

### AC-6: 保存・読込の後方互換性（既存データでの動作継続）

- **Given** 再編前の画面で保存した version 11 形式の config データが存在するアプリ
- **When** 再編後の設定画面を開く
- **Then** すべてのフィールド・ルール・ビュー個別設定が再編後の新しい配置に正しく反映される（値の欠落・誤反映がない）

### AC-7: 保存時の挙動が変化しない

- **Given** 再編後の画面で各セクションの値を編集する
- **When** 「保存」または「保存して更新」を実行する
- **Then** 保存される JSON の構造（キー名・`version: 11`）が再編前と完全に一致する（DOM 再編による保存データの形式変化がない）

## 6. 想定 UX / シーケンス

### 6.1 画面遷移フロー（変更なし）

再編は同一ページ内のセクション再配置のみであり、画面遷移・モーダル呼び出しのフロー自体に変更はない。

```
プラグイン設定画面を開く
  → init() 実行（loadInitialConfig → loadFields → loadStatuses
     → applyFieldMapping → applyPermissionRules → applyFieldValueRules
     → applyFilterConfig → デフォルト権限カラーピッカー挿入 → applyDefaultPermission
     → applySearchTargets → refreshPerViewSelect）
  → 管理者が各セクションを編集
  → 「保存」 or 「保存して更新」
     → collectFieldMapping / collectPermissionRules / collectFieldValueRules
        / collectDefaultPermission / collectSearchTargets / collectFilterConfig
        / collectViewConfig を呼び出し currentConfig を更新
     → setConfig（保存）/ 必要なら views.json PUT + deploy
```

### 6.2 管理者が新セクション構成で設定を行う際の体験（想定）

1. 「基本設定」でフィールドマッピングを完了する（必須3項目のみ集中）。
2. 「表示・絞り込み」で検索・フィルタ UI に出す項目を決める（ユーザー向け画面の見た目に直結する設定として認識できる）。
3. 「権限・色分け」で判定順序（フィールド→ユーザー→デフォルト）に沿って上から設定する。デフォルト権限設定が独立カードになっているため、「両方に一致しない場合のフォールバック」であることが見出しから直感的に分かる。
4. 「重複チェック設定」は独立セクションとして必要な場合のみ設定する（DnD 機能を使わないアプリでは無視できることが見出しから分かる）。
5. 「ビュー個別設定」で複数ビュー（部署別カレンダー等）ごとにタイトル・初期表示だけを設定する。この時点で「ここに来るまで権限・フィルタの設定をやり直す必要がない」ことが理解できる（従来は誤って「ビューごとに権限も変わるのでは」と誤解していたリスクの解消）。
