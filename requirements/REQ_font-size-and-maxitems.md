## 1. 背景・目的

### 1.1 課題 1: フォントサイズ縮小

ユーザーより「カレンダー全体の文字がやや大きい、一回り小さくしたい」との要望が確定。
現状は大部分のカレンダー要素（予定バー・chip・日付・曜日ヘッダー・ボタン等）が `font-size: 16px` の個別 px 指定で統一されているが、CSS カスタムプロパティ（変数）への一元化は行われていない。変更箇所が多いため、設計方式の選択が重要になる。

### 1.2 課題 2: ズームによる `_calcMaxItems` ずれ

ユーザーより「ブラウザの拡大率（ズーム）を変えると月ビューの `+N more` 表示判定がおかしくなる」との報告がある。`_calcMaxItems()` は `cellH` のみ `getBoundingClientRect()` で実測するが、その他の計算定数（`dateHeadH=32`, `moreH=20`, `itemH=27`）はすべて固定 px 値で定義されている。

### 1.3 両課題の連動性

フォントサイズを変更すると `moreH`（`+N more` 行高）・`dateHeadH`（日付行高）・`itemH`（chip/バー高）の実際の値が変化する。そのため、**課題 1 の実装時に課題 2 も同時に解決しておかないと、縮小後の定数ずれが新たな `_calcMaxItems` 誤差を生む**。両課題を同一 PR で対処することを強く推奨する。

### 2.2 `_calcMaxItems` の現状実装

`desktop.js:4812–4828`

```javascript
function _calcMaxItems() {
  var firstCell = _monthRoot ? _monthRoot.querySelector('.kc-month-cell') : null;
  if (!firstCell) return MAX_CELL_ITEMS;
  var cellH = firstCell.getBoundingClientRect().height;
  if (!cellH || cellH <= 0) return MAX_CELL_ITEMS;
  var dateHeadH = 32;        // dateline 占有高: min-height(28px) + padding-bottom(4px)
  var padding   = 4;         // セル padding 上下合計（CSS: padding 2px × 2）
  var moreH     = 20;        // +N more 行占有高（font 12px×lh + padding 2px + margin-top 1px ≈ 20px）
  var itemH     = BAR_H + BAR_GAP; // 1 件あたりの高さ（27px = 24 + 3）
  var available = cellH - dateHeadH - padding - moreH;
  if (available <= 0) return 0;
  var max = Math.floor(available / itemH);
  return Math.min(max, 10);
}
```

`BAR_H = 24` (`desktop.js:4790`), `BAR_GAP = 3` (`desktop.js:4791`) も固定定数。

### 2.3 ズームずれの理論的分析

#### 2.3.1 CSS px は「論理ピクセル」であり、ブラウザズームに対して本来は不変

CSS の `px` 単位は「CSS ピクセル（論理ピクセル）」であり、デバイスピクセル比（`devicePixelRatio`）には依存するが、**ブラウザの UI ズーム（Ctrl+/−）には通常追従する**。すなわち、ズームが 150% になると `16px` と指定した要素も描画上は「元の 1.5 倍の物理ピクセル数」で描かれるが、`getBoundingClientRect().height` が返す値は引き続き **論理ピクセル単位**（ズーム適用後の CSS ピクセル）のため、理論的にはズームで `cellH` も `dateHeadH`（実際の dateline 高）も同率で拡大・縮小し、割り算の結果は不変なはずである。

#### 2.3.2 実際のずれが発生する仮説

以下の要因により、理論上「不変」なはずの計算がずれる可能性がある。

**仮説 A: サブピクセル丸め（最有力）**

ズーム率が整数倍でない場合（例: 110%, 125%, 133%）、CSS レイアウトエンジンはピクセル境界を小数点以下で計算し、`getBoundingClientRect()` が小数値を返す。`Math.floor` で切り捨てるため、`cellH` の小数丸め方向によって計算結果が 1 ずれる。

例: ズーム 110% で `cellH=156px` が実際は `getBoundingClientRect().height = 155.99...` と返ると、`available = 155.99 - 32 - 4 - 20 = 99.99`、`Math.floor(99.99 / 27) = 3` のように 1 件分失われる。

**仮説 B: font-size のズーム時の実高変化**

固定 px 定数 `moreH=20` は「`font-size: 12px` × `line-height ≈ 1.4` + `padding 2px` + `margin-top 1px` ≈ 20px」という推算値であり、ズームやフォントヒンティングによって実際の行高が変わる（特に Windows の GDI フォントレンダリング）。

**仮説 C: `dateline` の実高**

`dateHeadH=32` は「`min-height: 28px` + `padding-bottom: 4px`」の合計だが、祝日名インライン表示（`desktop.css:884–892`）があると dateline が 28px 以上に膨らむ場合がある（ただし `min-height` で下限が決まっているため通常は 28px 固定）。ただし `flex` コンテナ内の `min-height` と `padding` の相互作用でブラウザ間差が出る可能性がある。

**結論**: 主因は「仮説 A（サブピクセル丸め）」と「仮説 B（moreH 推算値のずれ）」の複合である可能性が高い。ただし **根本的な解決は固定定数を実 DOM 計測に置き換えること**であり、定数を少し調整しても別のズーム倍率でずれが再発する。

### 2.4 `BAR_H` の散在状況（追加リスク）

`BAR_H = 24` は `desktop.js` 内で **4 箇所に独立して宣言**されている（行 2073, 3826, 4790, 5710）。これらはそれぞれ別スコープ（週ビュー・月ビュー各関数内）に定義されており、一元化されていない。フォントサイズ縮小で `BAR_H` も変更する場合、すべての箇所を同時に修正する必要がある。

なお、検索欄（`.kc-search-input` 等）は既に 14px のため実質変更なし（スコープに含めるが値は維持）。

#### FR-3: CSS カスタムプロパティによる一元管理（確定: 案 X1 採用）

`:root` に以下の変数を追加し、`font-size: 16px` を個別指定していた箇所をすべてこの変数参照に置き換える。

```css
:root {
  --kc-font-size-base: 14px;   /* カレンダーコンテンツ本体のベース文字サイズ */
  --kc-font-size-ui:   14px;   /* ヘッダー UI ボタン・ドロップダウン等 */
}
```

両変数の値は現時点で同一（14px）だが、将来個別調整できるよう分離して定義する。この方式により、将来の調整が `:root` の変数値変更のみで全要素に波及する。

#### FR-4: 行高・バー高の変更（Q-2 確定: BAR_H も縮小する）

フォントサイズを 14px に縮小するにあたり、**終日バー高（`BAR_H`）および chip 高・行高も文字サイズに整合するよう縮小する**。

- `BAR_H` の具体値（現状 24px）は builder が 14px フォントに対する視認性・padding を考慮して決定する
- CSS 変数 `--kc-ad-bar-h` の値も `BAR_H` の決定値に合わせて更新すること
- **`_calcMaxItems` の計算前提（`itemH` = `BAR_H + BAR_GAP` 等）も変更後の値に追従させること**（§4 FR-5 の実 DOM 計測方式を採用することで自動追従する）
- 「文字とバー高を整合させ、`_calcMaxItems` の前提も追従」を実装の必須要件とする

## 4. 課題 2: ズームによる `_calcMaxItems` ずれの要件

### 4.1 機能要件

#### FR-5: `_calcMaxItems` の固定定数を実 DOM 計測に置き換える（確定: 案 A 採用）

`dateHeadH`, `moreH`, `itemH` の 3 つの固定定数を、**実際の DOM 要素の高さを `getBoundingClientRect()` で計測して取得する方式**（案 A）に変更する。

これにより以下が達成される:

- ブラウザズーム（100%/110%/125%/150% 等）でのサブピクセル丸めずれが根治される
- FR-4 で `BAR_H` を変更した場合に `_calcMaxItems` が自動追従する
- 将来の CSS 変更（padding 調整等）にも自動追従する

詳細設計は §5 改善案比較の **案 A** を参照。

#### FR-6: `BAR_H` の一元化（確定: 今回スコープに含める）

`desktop.js` 内で 4 箇所に分散している `BAR_H = 24`（行 2073, 3826, 4790, 5710）を統一する。FR-4 で `BAR_H` の値が変更されるため、変更漏れ防止のために一元化は今回スコープとして扱う。

具体的な一元化手法（モジュールトップレベル定数化 or CSS 変数参照）は builder の判断に委ねる。ただし CSS `--kc-ad-bar-h` と JS 定数の二重管理は解消することが望ましい。

### 5.2 課題 2: `_calcMaxItems` 改善

#### 案 A: 実 DOM 計測ベース — **確定採用**

`_calcMaxItems` を呼ぶ前に、実際の DOM 要素（`.kc-month-dateline`、`.kc-month-more`、`.kc-month-chip`）の `getBoundingClientRect().height` を計測し、固定定数の代わりに使用する。

```
計測対象:
  - dateHeadH: firstCell.querySelector('.kc-month-dateline').getBoundingClientRect().height
  - moreH:     既存の .kc-month-more 要素があればその height、なければダミー要素を一時挿入して計測
  - itemH:     既存の .kc-month-chip 要素があればその height、なければ BAR_H + BAR_GAP のまま維持（フォールバック）
```

| 項目 | 評価 |
|---|---|
| 確実性 | 高（実際のレンダリング高を使うためズーム・フォント変化に完全追従） |
| 実装コスト | 中（計測タイミング管理が必要。ダミー要素の挿入は rAF 後に行う等） |
| リスク | 計測タイミングが DOM 更新前だと 0 が返る可能性（`placeMonthEvents` は `rAF` 後に呼ばれるため通常は問題なし）|
| ズーム対応 | 完全（どのズーム倍率でも正確） |

**詳細設計（実装時参照）:**

1. `_calcMaxItems` 冒頭で `firstCell` 取得後に dateline 要素を取得:
   ```
   var dateline = firstCell.querySelector('.kc-month-dateline');
   var dateHeadH = dateline ? dateline.getBoundingClientRect().height : 32;
   ```
2. `moreH` は `.kc-month-more` が存在する場合はその実高を取得:
   ```
   var moreEl = firstCell.querySelector('.kc-month-more');
   var moreH = moreEl ? moreEl.getBoundingClientRect().height : 20;
   ```
   既存の more 要素がない場合（初回 placeMonthEvents 前）のフォールバックは、CSS 変数から算出するか固定値 20 を維持する。
3. `itemH` は既存の chip があれば実測、なければ `BAR_H + BAR_GAP` をフォールバック。

#### 案 B: CSS 変数との同期

`:root` に `--kc-month-item-h: 27px`、`--kc-month-more-h: 20px` 等の変数を追加し、JS 側でこれらを `getComputedStyle(document.documentElement).getPropertyValue('--kc-month-item-h')` で参照する方式。

| 項目 | 評価 |
|---|---|
| 確実性 | 中（CSS 変数値が正確に管理されていれば追従できるが、変数更新漏れリスクがある）|
| 実装コスト | 中（CSS・JS 両側の変数管理が必要） |
| ズーム対応 | 部分的（CSS 変数値は論理 px 固定なのでサブピクセル丸めには対応不可） |

#### 案 C: ズーム変更時に再計算

`window.addEventListener('resize', ...)` で `window.devicePixelRatio` の変化を検知し、変化があれば `placeMonthEvents` を再実行する。

| 項目 | 評価 |
|---|---|
| 確実性 | 低（resize イベントはズーム変更で必ずしも発火しない。`devicePixelRatio` は CSS ズームでは変化しない場合がある） |
| 実装コスト | 低（既存の resize ハンドラに追記するだけ） |
| ズーム対応 | 不完全（ページロード後にズームしてもイベントが発火しないケースがある） |

**確定: 案 A（実 DOM 計測ベース）**

## 8. 想定 UX / シーケンス

### 8.1 フォントサイズ縮小による情報密度向上

変更前（16px ベース）の月ビューセルでは chip・終日バーが大きく、セル高 156px に対して最大 4〜5 件程度しか表示できない場合がある。**14px ベース**に縮小し、さらに FR-4 で BAR_H も詰めることで chip 高が下がり、同じセル高でより多くの予定を表示できる。ヘッダー UI（ボタン・ドロップダウン）も 14px に揃えることで、カレンダー全体の文字スケールが統一感のある見た目になる。

### 8.2 ズーム対応後の安定動作

実 DOM 計測により、ズーム倍率・フォントサイズ変更・将来の CSS 改訂に自動追従する。開発者が定数を手動で調整する必要がなくなる。
