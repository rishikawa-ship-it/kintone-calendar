## 1. 現状の全セクション/項目一覧（出現順）

`config.html` を頭から読んだ出現順。`<section class="kc-config-section">` は3個のみで、内部にサブブロック（カード風 `<div>`）が多数入れ子になっている。

## 2. セクションの並び順と論理グルーピングの評価

### 2.1 全体の並び順

```
[1] 共通設定（フィールドマッピング + 重複チェック設定）  ← アプリ全体
[1.5] 検索対象フィールド設定                            ← アプリ全体
[2] ビュー個別設定
    ├─ ①対象ビュー選択
    └─ ②選択中ビューの設定
        ├─ カレンダータイトル・初期ビュー              ← ビュー個別
        ├─ 権限フィールド設定                          ← アプリ全体（要素19）
        ├─ 権限ユーザー設定 + デフォルト設定           ← アプリ全体（要素20-21）
        └─ フィルタ設定                               ← アプリ全体（要素22）
```

### 2.2 論理グルーピングの評価

**問題点 1: 「ビュー個別設定」という見出しの中に、アプリ全体設定4種（権限フィールド・権限ユーザー・デフォルト権限・フィルタ）が閉じ込められている。**

- ユーザー視点では「ビュー個別設定というタイトルの下にあるから、ビューを切り替えると内容が変わる設定なのでは」と誤解しやすい。実際は権限・フィルタ設定はどのビューを選んでいてもフォームの内容は同一（`currentConfig.permissionRules` 等はビューIDに紐付かないグローバル状態）。
- config.html 内のコメント自体が「統合版: Phase 7 + Phase 9」（config.html:242）と記されており、履歴的にビュー個別設定セクションへ他機能が追加され続けた結果であることが読み取れる。
- 実装上も `applyPermissionRules()` / `applyFieldValueRules()` / `applyFilterConfig()` の呼び出しはビュー選択（`handleViewSelectChange`）とは独立した `init()` の初期化フローで1回だけ実行される（config.js:3671-3679）。ビュー切り替え時にこれらのテーブルが再描画される処理は存在しない。**すなわちコードの実装自体が「アプリ全体設定である」ことを裏付けている。**

**問題点 2: 「共通設定（フィールドマッピング）」セクションに、テーマの異なる3グループが同居している。**

- 「必須フィールド」（3項目）と「任意フィールド」（5項目）は同じ「フィールドマッピング」という主題でまとまっているため許容できるが、**「重複チェック設定」（判定モード + モードA/B条件分岐フィールド4項目）はフィールドマッピングとは異なる機能単位**であり、h3 サブ見出しで区切られているものの、セクション区切り（背景色・枠線カード）はない。DnD 移動・リサイズ時の重複ブロック機能というかなり独立した機能であるにもかかわらず、視覚的インパクトが弱い（h3 のみで、カード化されていない唯一の機能ブロック）。

**問題点 3: 検索対象フィールド設定（セクション1.5）が独立セクション化されている理由が構造上わかりにくい。**

- 検索対象フィールドも `fieldMapping` と同じ「アプリ全体でのフィールド関連設定」という点では共通設定セクションに含められそうだが、独立した `<section>` として分離されている。既存の `fieldMapping` オブジェクトとは別キー（`searchTargets`）で保存されるため実装上の都合はあるが、ユーザーから見れば「なぜ検索対象だけ別セクションで、権限系はビュー個別設定の中にあるのか」の基準が見えない。

**問題点 4: 権限フィールド設定 → 権限ユーザー設定 → デフォルト設定、という優先順位のある3層構造が、視覚的に対等な並びに見える。**

- `getPermission()` の実際の判定順は「①権限フィールド設定 → ②権限ユーザー設定 → ③デフォルト設定/フォールバック」（REQ_default-permission.md §2.1参照）。HTML 上も概ねこの順で並んでいるため大きな逸脱はないが、③のデフォルト設定が②のカード内部に埋め込まれているため、「デフォルト設定は権限ユーザー設定の一部機能なのか、それとも独立した第3のルールなのか」が見出し構造からは判別しにくい（h4 見出しがカードの内側にある）。

**問題点 5: フィルタ設定が「権限ユーザー設定」カードの外・同じ `.kc-view-edit-panel` 内に単独カードとして配置されている。**

- フィルタ設定は権限設定と直接の依存関係はない（フィルタのマッチ判定は独自ロジック）が、位置的には権限ユーザー設定の直後にあり、権限系3ブロック（フィールド/ユーザー/デフォルト）とフィルタ設定が同一階層で並ぶため、「権限」と「フィルタ」という異なる関心事が連続して並んでいる。REQ_checkbox-filter.md 内でも「関連性が高いため権限フィールド設定の直後に配置することを推奨する」との記述があり（§6.1）、設計側も両者に一定の関連性を見出しつつ、明確なグルーピング上の位置づけは定まっていない。

### 3.3 「フィールド選択→値選択」UI の重複度

同種の「対象フィールドを選ぶ→そのフィールドの値/選択肢を選ぶ」という2段階UIパターンが、**独立した実装として最低3箇所**存在する。

| UI箇所 | フィールド選択関数 | 値選択関数 | 対象フィールド型 |
|---|---|---|---|
| 権限フィールド設定 | `buildFieldValueFieldSelect()`（config.js:1488） | `buildFieldValueValueSelect()` + `rebuildValueSelect()`（config.js:1381, 1536） | DROP_DOWN/RADIO_BUTTON/CHECK_BOX/STATUS |
| フィルタ設定（fieldValue型グループ） | `buildFilterTargetFieldSelect()`（推定。config.js:1720行以降のフィルタ設定UI群） | `getFieldValueOptions()` を再利用した候補選択 + 手動追加行 | DROP_DOWN/RADIO_BUTTON/CHECK_BOX/STATUS/USER_SELECT |
| 権限ユーザー設定 | `buildPermissionFieldSelect()`（config.js:1233） | （値選択なし。フィールド選択のみで対象は USER_SELECT 固定） | USER_SELECT |
| 検索対象フィールド設定 | `buildSearchTargetRow()` 内 select 生成（config.js:2263）| （値選択なし。フィールド選択のみ） | SINGLE_LINE_TEXT/MULTI_LINE_TEXT/RICH_TEXT/REFERENCE_TABLE/USER_SELECT |

**重複の実態**:
- `getFieldValueOptions(fieldCode, fieldType)`（config.js:1360）と `rebuildValueSelect()`（config.js:1381）は権限フィールド設定用に実装された関数だが、REQ_checkbox-filter.md の設計（§6.4）でフィルタ設定にも「そのまま呼び出し」で流用する方針が明記されている。**値取得ロジックそのものの再利用は意図的に行われている**一方、行DOM生成（`buildFieldValueRow` / `buildFilterGroupRow` / `buildFilterItemRow`）・DOM からの収集（`collectFieldValueRules` / `collectFilterConfig`）・DOM への反映（`applyFieldValueRules` / `applyFilterConfig`）は、それぞれ独立した関数として個別実装されている。
- ドラッグハンドル（`buildDragHandle()`）・行DnD（`attachRowDragEvents()`）・削除ボタンスタイル（`.kc-permission-del-btn` / `.kc-fieldvalue-del-btn` / `.kc-search-target-delete` — 3つとも同一 CSS 定義の重複コピー、config.css:762-788, 918-945, 1021-1048）は関数レベルでは共通化されている箇所（DnD）と、CSSレベルで丸ごとコピーされている箇所（削除ボタン）が混在する。
- カラーピッカーウィジェット（`buildColorPickerWidget()`）は唯一、権限フィールド設定・権限ユーザー設定・デフォルト設定の3箇所すべてで完全に共通化されている（好例）。

**評価**: 「テーブル型の行UI」という骨格は5箇所（権限フィールド/権限ユーザー/検索対象/フィルタグループ/フィルタ項目）で類似パターンを再発明しており、共通コンポーネント化の余地が大きい。カラーピッカーのように完全共通化できた部品もあれば、行構築・収集・反映の3関数セットは毎回individually実装されている。

## 4. 一貫性/重複/わかりにくさの評価

### 4.1 用語・ラベルの一貫性

| 観点 | 評価 |
|---|---|
| セクション見出しの命名パターン | 「〇〇設定」で統一されている（共通設定/検索対象フィールド設定/ビュー個別設定/権限フィールド設定/権限ユーザー設定/フィルタ設定）。一貫性は高い |
| 「フィールド」という語の多義性 | 「フィールドマッピング」「権限フィールド設定」「検索対象フィールド設定」「重複判定キーフィールド」等、"フィールド"の語が指す対象が毎回異なる（kintoneフィールドそのもの、判定対象フィールド、識別キー用フィールド）。文脈で判別可能だが、初見の管理者には区別しにくい可能性がある |
| 「権限」という語の使い分け | 「権限フィールド設定」「権限ユーザー設定」「デフォルト設定（権限）」の3つで用途が異なる判定条件（値マッチ/ユーザーマッチ/非マッチ時）を指す。REQ_edit-permission-extension.md で「編集権限設定」から「権限ユーザー設定」へリネームされた経緯があり（config.js コメント冒頭 v6変更点）、命名の反復的な見直しが行われてきている |
| ボタンラベルの一貫性 | 「+ 行を追加」（権限フィールド/権限ユーザー/検索対象）、「+ グループを追加」（フィルタ）で表記が微妙に異なるが、対象が「行」か「グループ」かの違いを反映しており妥当 |

### 4.2 必須/任意の区別

- `<span class="kc-required">*</span>` による必須マーカーは**セクション1（共通設定）の3項目（タイトル/開始/終了）のみ**に付与されている（config.html:43, 53, 63）。
- 他のセクション（検索対象・権限系・フィルタ）には必須/任意の視覚的区別が一切ない。これらは「0件でも動作する」設計（例: 権限ユーザー設定は0件で全員編集可、REQ_edit-permission-extension.md 記載）のため、必須マーカーが不要なのは仕様として妥当だが、**「このセクションは空でもよい」という前提がヒントテキストでしか説明されておらず、視覚的な手がかり（例: バッジ表示）がない**。

### 4.3 初期値・プレースホルダ・ヘルプ文

| 項目 | プレースホルダ | ヘルプ文（`.kc-config-hint`） |
|---|---|---|
| 必須3フィールド | なし（`-- フィールドを選択 --`のみ） | 対象型を明記（あり） |
| 終日ラベル値 | `例: 終日`（あり） | あり |
| カレンダータイトル | `例: 会議室予約カレンダー`（あり） | あり |
| 権限フィールド設定 | — | あり（優先順位の説明あり） |
| 権限ユーザー設定 | — | あり（0件時の挙動を明記） |
| デフォルト設定 | — | あり（チェックオフ時の挙動を明記） |
| フィルタ設定 | — | あり（OR/AND・全解除の意味を明記） |
| 検索対象フィールド設定 | — | なし（`kc-search-target-section` に `.kc-hint` クラスの記述はCSSにあるが、config.html 上に該当する `<p>` 要素は見当たらない） |

**評価**: 全体的にヘルプ文の付与率は高く、特に新しく追加されたセクション（権限フィールド/権限ユーザー/デフォルト/フィルタ）は判定ロジックの複雑さ（優先順位・OR/AND・0件時挙動）を丁寧に文章化している。ただし検索対象フィールド設定にはヘルプ文がなく、記述の手厚さにセクション間でばらつきがある。

### 4.4 依存関係の可視化

| 依存関係 | 画面上の表現 | 評価 |
|---|---|---|
| 重複チェック: 判定モードにより表示エリアが切替 | `#kc-overlap-mode-a-area` / `#kc-overlap-mode-b-area` を `style="display:none"` で初期非表示、ラジオ変更で表示切替（config.js:3819-3827） | わかりやすい（条件付き表示がラジオ直下に出る） |
| メールアドレス初期値チェックボックス: フィールド未選択時は disabled | `updateMailCheckboxState()` で連動 | 挙動は実装されているが、disabled 理由がツールチップ等では説明されていない |
| デフォルト設定: チェックボックスOFF時に select/カラーピッカーが disabled + opacity 0.5 | `_syncDefaultPermissionEnabled()` | わかりやすい（視覚的にグレーアウト） |
| フィルタ設定: 「担当」グループが削除不可 | HTML上の記述は不明（config.html の静的部分に削除不可バッジ等の記載は見当たらず、config.js 側のグループ行生成ロジックで制御されている可能性。config.html 側では確認できず） | **画面全体を見ないと「なぜ担当グループに削除ボタンがないのか」がわからない可能性がある**（要 config.js 該当箇所の追加確認） |
| ビュー個別設定内の権限/フィルタ系がビューを切り替えても内容が変わらない | 画面上の表現なし | **§2.2 で述べた最大の構造課題。ユーザーが誤解しやすい** |
| 権限フィールド設定 → 権限ユーザー設定 → デフォルト設定の優先順位 | ヒント文で説明されているが、視覚的な「上から順に評価される」という表現（例: 矢印・番号）はない | テキストのみで補われている |

### 4.5 レイアウト課題

- **縦の長さ**: セクション2（ビュー個別設定）が全体の大半を占め、フィールドマッピング関連（17-18）+ 権限フィールド設定テーブル（可変長）+ 権限ユーザー設定テーブル（可変長）+ デフォルト設定 + フィルタ設定（グループ×項目の二重可変長リスト）が単一スクロール領域に縦積みされる。フィールド数・ルール数・グループ数が増えるほど、目的の設定にたどり着くまでのスクロール量が線形に増加する。
- **グループ見出しの強弱**: `<h2>`（3個）> `<h3>`（5個程度）> `<h4>`（1個、デフォルト設定のみ）という階層だが、視覚的な強調（背景色付きカード化）は `.kc-permission-section` / `.kc-fieldvalue-section` / `.kc-filter-config-section` / `.kc-view-control-panel` の4箇所にのみ適用されており、「共通設定」セクション内の重複チェック設定や、検索対象フィールド設定にはカード化がない。**同じ見出しレベル（h3）でも視覚的な重みが不揃い**。
- **カード内カード**: デフォルト設定は権限ユーザー設定カードの内部にネストされたカード（`border-top` で区切り）になっており、「カードの中にカードがある」構造は他のセクションにない特殊パターン。

### 4.6 全画面でないと使いにくい可能性

- 権限フィールド設定・権限ユーザー設定のグリッドは6-7列（ハンドル/フィールド/値/権限/背景/文字/削除）を `grid-template-columns: 24px 2fr 1.5fr 1fr 44px 44px 40px`（config.css:833）で配置しており、`max-width: 800px`（config.css:30, コンテナ全体の最大幅）という制約下では、特に「フィールド」列（2fr）が長いラベル（例: `フィールド名 (fieldCode)` 形式、config.js:1502 で `label + ' (' + code + ')'` 表示）で圧迫されやすい。
- フィルタ設定の項目行はさらに列数が多く5列（`grid-template-columns: 20px minmax(0, 1fr) minmax(0, 1fr) minmax(80px, 90px) minmax(36px, 40px)`, config.css:1478）で、コメントに「2026-07 実機フィードバック: 初期ONトグルが最終行で見切れる不具合の再発防止」との記載があり（config.css:1474-1475）、**実際に狭幅での見切れ不具合が過去に発生・修正された実績がある**。iframe 埋め込みの管理画面という前提上、画面幅が保証されないため、今後も列数の多いテーブルUIは同種の問題を再発しやすい。
- レスポンシブ対応（`@media (max-width: 600px)`, config.css:1224-1255）は共通設定セクションの基本要素（input/select/radio）のみを対象にしており、権限テーブル・フィルタテーブルの grid レイアウトはブレークポイント対応の記述がない。

## 5. 今セッションで増えた項目の収まり

今セッションで追加された2機能（デフォルト権限設定・チェックボックス式フィルタ設定）の収まりを評価する。

### 5.1 デフォルト権限設定（REQ_default-permission）

- **配置**: 権限ユーザー設定カードの内部最下部（`+ 行を追加` ボタンの直後）に、`border-top` で視覚的に区切ったサブブロックとして追加された（config.html:401-433）。
- **収まりの評価**: 判定順序（フィールド → ユーザー → デフォルト）にほぼ沿った位置であり、機能的な文脈は破綻していない。ヒント文で「上のいずれの条件にも一致しない場合」と明記され、依存関係の説明も丁寧。
- **課題**: 「権限ユーザー設定」というカードの**内部**に置かれているため、見出し階層上は権限ユーザー設定の一部に見えるが、実際は権限フィールド設定・権限ユーザー設定の**両方**を通じたフォールバック設定である（REQ_default-permission.md §1.1で「両テーブルを通じて」と明記）。**この「両方にかかる」という性質が、片方のカード内部に置かれたことで伝わりにくい**。カード外・両テーブルの下に横断的なブロックとして配置する方が構造的には正確だった可能性がある。

### 5.2 チェックボックス式フィルタ設定（REQ_checkbox-filter）

- **配置**: 権限ユーザー設定カード（デフォルト設定を含む）の直後に、独立した新規カードとして追加された（config.html:440-458）。
- **収まりの評価**: 独立カードとして視覚的な区切りが明確であり、デフォルト権限設定より構造的には健全（カード内カードになっていない）。DnD・行追加・グリッドレイアウトのパターンも既存のテーブル型UIと統一感がある。
- **課題**: 権限設定とフィルタ設定は判定ロジック上の直接的な依存関係はない（フィルタは `getPermission().bgColor` を「担当」グループの判定にのみ間接利用）にもかかわらず、両方とも「ビュー個別設定」という見出しの中、かつ権限設定の直後という位置にあるため、**「権限」と「フィルタ」という別テーマが連続配置されている**（§2.2 問題点5と同一）。
- 「新セクションを増やすたびに `.kc-view-edit-panel` の最後に追記する」という増築パターンが今回も踏襲されており（config.html コメント「統合版: Phase 7 + Phase 9」の系譜が continue している）、次に新機能が追加されればさらに縦に伸びることが予想される。

### 5.3 総合評価

両機能とも個別の実装品質（ヒント文・disabled制御・DnD対応）は高いが、**「どこに置くか」の判断基準がプロジェクト全体で明文化されておらず、直近のセクションの下に追記する」という運用が続いている**。これは§3.2で確認したマイグレーション履歴（v6以降のほぼ全バージョンが新規追記型）とも整合する、プロジェクトのこれまでの成長パターンそのものである。今回の2機能が「悪い」わけではなく、**この増築パターンの限界が今回で可視化された**と捉えるのが妥当。

## 6. 改善の方向性（骨子・3案）

以下は詳細設計ではなく、次段の設計検討のための選択肢提示（骨子のみ）。

### 案A: セクション再編（グルーピングの正規化）+ 現行の単一スクロール構成を維持

**内容**: HTML構造・見出し階層を、実際のデータ粒度（アプリ全体設定 / ビュー個別設定）に合わせて再編する。具体的には「ビュー個別設定」セクションから権限4ブロック（フィールド/ユーザー/デフォルト/フィルタ）を分離し、「アプリ全体のルール設定」という新セクションとして独立させる。ビュー個別設定セクションは「カレンダータイトル・初期ビュー」のみのシンプルなセクションに戻す。

| メリット | デメリット |
|---|---|
| データの実態と画面構造が一致し、「ビューを切り替えても権限設定は変わらない」という誤解が解消される | セクション数が増える（現状3 → 4程度）。縦の長さ自体は変わらない（スクロール量は同等） |
| 実装コストが比較的小さい（HTML の移動 + 見出しラベル変更が主。config.js の DOM 取得元IDは変えずに済む） | 権限フィールド設定とフィルタ設定の関係性（§2.2問題点5）は解消されない |
| 既存のテーブルUI・DnD実装への変更は不要 | 「アプリ全体のルール設定」という新セクション自体がさらに縦長になる可能性が残る |

### 案B: タブ or アコーディオンによる情報設計の刷新

**内容**: 現行の縦積み単一ページ構成を、タブ（例: 「基本設定」「表示設定」「権限設定」「フィルタ設定」「ビュー管理」）またはアコーディオン（各カードを折りたたみ可能にし、デフォルトは主要セクションのみ展開）に変更する。

| メリット | デメリット |
|---|---|
| 縦スクロール量が大幅に減り、目的の設定に直接アクセスしやすくなる | 実装コストが大きい（タブ切替のJS制御、フォーム送信時に非表示タブの値も収集する必要があるため `saveConfig()` の収集ロジックに影響なし設計が必要） |
| 関心事ごとに画面を分離できるため、§2.2の「異質なテーマの混在」問題を根本的に解消しやすい | kintone 管理画面の iframe という制約下でタブ内スクロール・タブ外レイアウトの検証コストが増える（狭幅環境での見切れ再発リスク、§4.6参照） |
| 将来の機能追加時も「どのタブに置くか」で位置決めの基準が明確になる | 初見の管理者が「他のタブに何があるか」を発見しにくくなる可能性（発見性の低下） |

### 案C: 共通コンポーネント化 + セクション構造は現状維持（内部リファクタリング優先）

**内容**: 見た目のセクション構成はほぼ変えず、§3.3で指摘した「フィールド選択→値選択」テーブルUIパターン（権限フィールド/権限ユーザー/検索対象/フィルタグループ/フィルタ項目の5箇所）を共通コンポーネント（行ビルダー・収集・反映の3関数セット）に統合する。削除ボタン・ドラッグハンドルのCSSも共通クラスに一本化する。

| メリット | デメリット |
|---|---|
| ユーザー向けの画面構成は変わらないため、operator（既存ユーザー）の混乱が最小限 | ユーザーが「整理したい」と感じている見た目の課題（縦の長さ・グルーピング）そのものは解決されない |
| 保守性が大きく向上する（今後新しい「値ベースの権限/フィルタ系」機能が追加されるたびに個別実装するコストが下がる） | 見た目に現れる改善効果が小さく、ユーザー要望に対する直接的な回答にならない可能性 |
| リスクが低い（内部実装のみの変更なので、既存の保存データ形式・migrationに影響を与えずに実施できる） | 案A/Bの前提作業として位置づけることもできるが、単独では「整理」というユーザー要望を満たしきれない |

### 案の組み合わせについての所感

3案は排他的ではない。**案C（共通化）を基盤にしつつ、案A（グルーピング正規化）で構造を正し、必要であれば将来的に案B（タブ化）へ発展させる、という段階的な適用が現実的**と考えられる。特に案Cは案A・Bのどちらを選んでもリファクタリング効果が活かせるため、着手順序としては最も低リスクである。次段の設計フェーズで、ユーザーの優先度（見た目の即時改善 vs 保守性）を確認して詳細設計に進むことを推奨する。
