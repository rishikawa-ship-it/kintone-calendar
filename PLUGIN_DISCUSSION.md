# kintone プラグイン化検討 -- 設計担当 x 管理者 議論まとめ

## 結論

**プラグイン化は技術的に可能であり、中長期的に推奨。**
現在の KC_ プレフィックス方式でも動作するため「今すぐ」必須ではないが、複数アプリ展開や非エンジニアへの設定委譲が必要になった時点で着手が最適。

移行工数: **中規模（10-17営業日）**

---

## 設計担当 ↔ 管理者の議論

### 論点1: プラグイン化で何が変わるか

**管理者**: KC_プレフィックス方式で動いている。プラグインにするメリットは？

**設計担当**: 現在ハードコードされている設定がGUIで変更可能になる:
- `EXCLUDED_STATUSES: ['返却済', '削除済']`
- `ALLDAY_LABEL: '終日'`
- ヘッダータイトル `'貸与Wi-Fi予約カレンダー'`
- ステータス選択肢 `予約済/貸出中/返却済`

さらに、フィールドマッピングをドロップダウンで選択でき、KC_プレフィックス規約に縛られなくなる。

### 論点2: 複数アプリへの展開

**管理者**: 他部署でも同様のカレンダーを使いたい要望がある。

**設計担当**: プラグインなら:
- kintoneシステム管理で1回インストール
- 各アプリの「プラグインの設定」で個別にフィールドマッピング
- JS/CSSファイルの個別管理が不要
- アップデート時もZIP 1回で全アプリに反映

### 論点3: 設定画面のUX

**管理者**: 非エンジニアでも設定できるか？

**設計担当**: kintone APIでフィールド一覧を取得し、ドロップダウンで選択させる実装にすれば可能。kintone公式プラグインテンプレートにもサンプルがある。

### 論点4: KC_プレフィックス方式との両立

**管理者**: 既存のKC_プレフィックス方式は残せるか？

**設計担当**: 可能。以下の優先順位で解決:
1. プラグイン設定に値があれば → それを使用
2. なければ → KC_プレフィックスで自動検出（従来互換）
3. どちらもなければ → エラー

### 論点5: カスタマイズビューHTMLの制約

**管理者**: `<div id="kc-root">` の設置は自動化できるか？

**設計担当**: kintoneのカスタマイズビューHTML設定はAPIでは変更不可。プラグインでも手動設置が必要。ただし、プラグインの設定画面に手順ガイドを表示することは可能。

### desktop.js の主要変更点

```javascript
// 現在: (function() { ... })();
// 変更後:
(function(PLUGIN_ID) {
  // プラグイン設定を読み込み
  var conf = kintone.plugin.app.getConfig(PLUGIN_ID);
  if (conf && conf.fieldTitle) {
    KC.Config.FIELD.title = conf.fieldTitle;
    KC.Config.FIELD.start = conf.fieldStart;
    KC.Config.FIELD.end   = conf.fieldEnd;
    // ...
  } else {
    // フォールバック: KC_プレフィックス自動検出
    await KC.Config.detectFields();
  }
})(kintone.$PLUGIN_ID);
```

## メリット・デメリット比較

| 観点 | 現在（カスタマイズJS） | プラグイン化後 |
|---|---|---|
| 設定変更 | JSファイル直接編集 | GUI設定画面 |
| 複数アプリ展開 | アプリごとにJS/CSSアップロード | 1回インストール + アプリごとに設定 |
| アップデート | 全アプリのJS/CSSを個別更新 | ZIP 1回で全アプリ反映 |
| フィールドマッピング | KC_プレフィックス規約が必要 | ドロップダウンで自由選択 |
| 非エンジニア運用 | 不可 | 可能 |
| 初期構築コスト | 低（完成済み） | 中（+10-17日） |
| 保守コスト | 中（変更のたびにコード修正） | 低（GUI変更） |

### 推奨判断

| シナリオ | 推奨 |
|---------|------|
| 1アプリのみ・設定変更少 | 現状維持 |
| 2アプリ以上に展開予定 | プラグイン化推奨 |
| 非エンジニアに設定委譲 | プラグイン化を強く推奨 |

## 将来検討項目

### 組織選択 / グループ選択フィールド連携

#### 概要

ユーザー選択 (`USER_SELECT`) フィールドとログインユーザーの照合は直接比較で実装可能（既に検討済）。一方、組織選択 / グループ選択フィールドの照合はログインユーザーの所属情報を取得する追加 API が必要なため、**プラグイン化フェーズでの対応検討**とする。

想定用途:

- 「自分の組織の予定のみ表示」フィルタ
- 組織別ハイライト / 色分け
- アクセス制御（他組織の予定は読み取り専用）

#### 必要 API（cybozu.com 共通 API）

| API | 用途 |
|-----|------|
| `kintone.api('/v1/users/organizations.json', 'GET', {code: loginUser.code})` | ログインユーザーの所属組織一覧を取得 |
| `kintone.api('/v1/users/groups.json', 'GET', {code: loginUser.code})` | ログインユーザーの所属グループ一覧を取得 |

#### 実装パターン案

- 起動時 1 回取得 + メモリキャッシュ（操作中は追加 API 呼び出しなし）
- 各レコードの組織 / グループフィールド値と `some` / `includes` 等で照合

#### 想定する活用例

1. 「自分の組織の予定のみ表示」フィルタトグル
2. 「自分の所属グループの予定」のハイライト表示
3. 組織別カラー分け
4. 「他組織の予定は読み取り専用」アクセス制御

#### 制約・注意点

| 項目 | 内容 |
|------|------|
| API 権限 | cybozu.com 共通 API はログインユーザー自身の情報なら閲覧可。他ユーザー情報取得は admin 権限が必要 |
| API 呼び出し回数 | 起動時 +2 回（org + group）。キャッシュにより操作中の追加負荷はなし |
| 多階層組織 | 親組織 / 子組織の継承関係は API 戻り値に含まれない（要別途検討） |
| API 失敗時のフォールバック | 全表示 / 表示なし のいずれとするか未決定 |

#### プラグイン化フェーズで対応する理由

- 基本機能と独立した拡張機能であり、現フェーズに含める必然性がない
- アプリ単位で有効化 / 無効化したい
- プラグイン設定画面で「組織フィールドコード」「グループフィールドコード」をドロップダウン選択させる UI が必要
